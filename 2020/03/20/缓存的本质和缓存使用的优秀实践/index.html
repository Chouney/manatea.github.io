缓存的本质和缓存使用的优秀实践

发表于 2020-03-20 更新于 2020-03-23

缓存特性Tips小记

因为书中关于缓存的知识原理性的东西比较多，因此这里仅列出一些自己认为比较重要的点当作tips记录

一.CPU的缓存架构

图一：CPU缓存架构 。
L1、L2为核级别的缓存，L3为多个核共享的缓存。

  • cacheline 缓存单元：缓存操作的最小单元，通过内存单元的物理地址来访问缓存。缓存在使用时是通过类似二位数组的形式来存取，我们常看到的8WAY、16WAY这种单位是用来描述缓存，通常用WAY表示列，SET表示行，通过防卫物理地址的高位和中位来选中一个缓存单元。

    通常的缓存单元大小为64个字节。

  • 伪共享：指的其实是多个线程如果读写一个cacheline里面的不同变量，会出现资源竞争而导致性能下降；或者是单线程每次的读写请求都没有命中cacheline；

二.分布式的缓存架构

与CPU缓存架构相似，我们基本可以得出一个与其相似的分布式缓存架构出来。因此我们会面对于CPU缓存架构相同的问题缓存高可用、缓存高并发。

2.1 缓存高可用

目前主流的模式是基于分片的异步复制的主从模型。将数据按哈希环进行分片，然后针对不同分片还存在主从副本，同时异步进行主从同步。

如果需要对缓存的高可用也可以通过强一致性协议来完成，即：

  1. 主开启写事务
  2. 主分发给子开启写事务
  3. 子写完返回成功给主
  4. 主收到所有子完成返回成功

redis的高可用是通过在每一个虚拟机节点上配置一个哨兵Sentinel，通过哨兵监控进行心跳检测、更新及选举。

2.2 缓存高并发

缓存理论上不存在高并发问题，问题一般都是出在IO成本上。

2.3 Redis和Memcache两个解决方案的对比

  • 线程模型
    • redis是单线程的非阻塞IO模型，适合快速的短逻辑操作。针对大量长逻辑的场景时应当考虑配置多个实例来提高CPU利用率，官方推荐的是1台机器8个实例。此外为了提高单线程的IO效率，还提供了管道来合并请求和执行批量操作。依赖libevent库的两个epoll文件和自己实现了简单的事件通知模型
    • memcache是多线程的非阻塞IO模型，直接依赖了libevent库。
  • 持久机制
    • redis支持RDB和AOF，基于定时的持久和基于操作的持久化。memcache依赖第三方组件
  • 客户端
    • redis的常见客户端jedis使用的是阻塞I/O，但可以配置线程池且支持一致性哈希分片。

      非阻塞I/O模型会有一个额外的请求线程（BOSS线程池），用来监听端口和处理就绪数据。工作者线程则用来计算。

  • 高可用
    • Redis支持节点复制，支持故障恢复和集群方案；memcache依赖第三方组件。
  • 事务
    • redis支持了一些线程安全和事务的命令，同时因为redis是单线程的，所以所有在服务器执行的操作都是原子性的。但是跨客户端的操作并不能保证原子性。
  • 内存分配
    • Redis 屏蔽了不同平台之间的差异，对内存分配函数封装了一层，程序中统一使用”zmalloc”、”zfree”系列函数，位于zmalloc.h/zmalloc.c文件中：
      • 若系统中存在google的TC_MALLOC库，则使用tc_malloc系列函数代替原本的malloc函数
      • 如果是Mac系统，则使用系统的内存分配函数。
      • 其他情况对于每个分配好的空间前加一个定长字段来记录分配空间大小

2.4 应用层的缓存应用模式

双读双写

比较好理解，读即先读缓存，未命中则再读库后写缓存；写即先写库再写缓存。

异步更新

这种模式是基于缓存的核心架构，所有数据全量不过其写缓存，并通过定时异步读取数据库中的变更数据更新到缓存中。

定时异步或者是每次更新时异步读mysql的binlog来实现缓存的更新（或者是其他存储引擎，比如es；扩展一些，可以基于dataBus数据总线的概念，将所有db库的更新通过binlog推送至其他存储引擎中）

串联代理

通过在应用和数据库中添加一层缓存代理组件，所有的读写操作通过代理组件做数据库的更新及维护，如Varnish缓存加速。

2.5 缓存分片的三种方式

这个非常类似之前的数据库分片方式

  • 客户端分片，通过扩展jar包实现分片解决方案
  • 代理分片，常见的有Codis框架
  • 集群分片，Redis3.0，同时集群可以实现高可用性。

2.6缓存迁移

分别为平滑迁移，停机迁移以及一致性hash环。其中平滑迁移方案同前一张的db方案

2.7 缓存穿透、缓存并发和缓存雪崩

上述三个点是缓存应用场景中常见的

缓存穿透

穿透的含义就是由于大量请求未命中缓存导致请求直接访问到数据库中。

预防措施：可以通过校验请求参数key的合法性，来过滤掉一部分的恶意供给

缓存并发

缓存并发就是大量对同一key值请求的同时,key值缓存失效，造成大量的并发读库写缓存的操作。

预防措施：软过期。保存一个缓存失效时间，可以派遣一个单独的线程从业务层面上去判断缓存是否过期，如果快过期则提前更新缓存。

缓存雪崩

原因一般都是缓存服务器重启，或者缓存在同一时间内失效。会造成这一时间段内数据库压力大量上升。

预防措施：通过对不同的缓存设置不同的失效时间，将失效时间分散。

3.2 一些优秀的redis案例tips

  • 一定要加缓存监控，方便后续慢查询、大对象、内存使用请款问题的排查。
  • 任何缓存一定要设置失效时间，同时失效时间必须不能在同一时间内。
  • 访问缓存的连接一定要设置超时时间，避免负载上升后连接池超大从而雪崩的情况。
  • 低频的访问不要用缓存。
  • 缓存数据不能太大。
  • 对于value值较多的key，不推荐使用hgetall方法，会影响其他请求的访问。
  • 写缓存一定要写正确的数据，避免任何空指针，程序异常等。
  • 使用缓存的时候一定要有降级备案，尤其对核心业务。
