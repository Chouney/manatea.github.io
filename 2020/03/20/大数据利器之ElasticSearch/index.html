大数据利器之ElasticSearch

发表于 2020-03-20 更新于 2020-03-23

Lucene

在认识es之前，需要先对lucene有一个整体的认识

一. lucene

1.1 核心模块

lucene的核心模块如图所示：

analysis: 词法分析，主要是做分词
index：创建索引
store：负责索引的读写等文件操作
queryParser：查询语句的语法分析器
search：负责搜索索引
similary：计算相关性打分和排序

1.2 核心概念

  • Term：最小分词单位

  • Term Dictionary：Term的集合，数据结构一般有数组、hash、以及fst(finite-state transducer)

  • 倒排表：存储结构

  • 段 segment：索引的最小独立存储单元。段具有不变性，即一旦生成，不可写只能读。

1.3 检索方式

词的检索方式主要有，AND、OR、NOT。大致的查询方式就是先根据Term1查询倒排表，拿到文档集合1；在通过运算方式查询Term2，拿到文档集合2后再做归并。

1.4 分段存储

分段存储是为了减轻写操作对索引变更的影响，同时分段采用不可变的设计，是为了提高读写性能，不用锁，也不用考虑多线程读写一致性问题。

我们分别阐述一下各个操作是如何影响分段的：

  • 新增，在新的段中添加新的索引
  • 删除，由于段不可变，所以在删除索引的时候通过添加一个标记文件.del，里面记录删除索引的id。在后面归并段的时候才会真正删除。因此被删除的段是可以被查询到的，只是在做合并的时候才被过滤调。
  • 更新，由于不可变，所以更新操作需要删除旧的索引后新增新的索引。因此在考虑为数据添加进索引的使用应该优先考虑数据的更新频率，太高的更新频率会导致大量空间浪费。

除此之外，我们还应该知道段是被延迟写入的，内容会先写到内存，然后通过时间间隔或者容量阈值来做异步刷新，生成提交点。

在这里，段被写在内存是只写非读的，这点与在磁盘中是相反的。所以会存在准实时的延迟问题。
这里es通过事务日志来保证提交数据的安全。

1.5 段合并策略

lucene的段合并的策略是：先将段按大小分组，对同组进行合并。此外当段大于某一阈值时，便不在做合并。

具体的段合并算法过程讲解，可以自行参考博客园刘超绝先的博客

1.6 Lucene的打分机制

lucene的相似度因子：

  • tf:某个term在文档中出现的次数

$$tf = \sqrt{termCountInDoc}$$

  • idf:表示的是整个文档集合中有这个词的文档数量越少越重要。
    $$idf = 1 + Log(\frac{docCount}{docFreq+1})$$

其中docCount为全量文档集合，docFreq指出现该term的文档数

  • length: 表示的是同等条件下，term所在文档的词量越小，相似度就应该越高。
1.6.1 基于向量空间模型

核心思想是把文本信息映射到向量空间中，并通过计算向量之间的夹角来得出相似度。

向量分为查询语句向量，和文档向量，向量指的是文档的权重，即文档内的term权重集，其中权重即tf x idf。

关于具体的score相似度推导过程，可以自行查询。这里只列出结果

$$score(q,d) = \frac{1}{\sqrt{\sum_{t=0}^{t=f}{idf(t^2)}}} \times \sum({tf(tind) \times idf(t^2) \times \frac{1}{\sqrt{numOfTermsInFieldf}}})$$

加入相似度打分公式后：

$$score(q,d) = coord(q,d) \times queryNorm(q) \times \sum{tf(tind) \times idf(t^2) \times t.getBoost() \times norm(t,d)}$$

其中

coord(q,d)= 文档d中包含查询词的个数/查询语句中包含t的个数

$$queryNorm(q) = \frac{1}{\sqrt{q.getBoost()^2 \times \sum{(idf(t) \times t.getBoost()^2)}}}$$

$$norm(t,d) = d.getBoost() \times lengthNorm(field) \times \prod_{fieldFInd}{f.getBoost()}$$

1.6.2 基于概率的模型

BM25算法是根据BIM(Binary independent Model 二元独立模型)算法改进而来的，二元独立模型存在两个假设：

  • 一个词和文档的关系只有，相关和不相关两种关系
  • 文档里的词与词之间没有任何关联。

BM25算法在BIM基础上添加了词的权重和两个经验参数，elaschsearch的默认算法已经由先前默认的向量空间模型变成了BM25。

BM25的计算公式主要分两部分：

$$socre(Q,d) = \sum_{i}^{n}{W_i \times R(q_i,d)} = \sum_{i}^{n}{IDF(q_i) \times \frac{f_i \times (k_1 + 1)}{f_i+k_1 \times (1-b+b \times \frac{dl}{avgdl})}}$$

其中W是第词的权重，R(q,d)是每个词和文档的相关度值。

这里的IDF值的计算方式与上述不太相同，但最后表示的含义都是一样的。

f：值词在文档中出现的次数

dl：表示文档的长度

avgdl：文档平均长度

k1:调节因子，调整词频对得分的影响，默认1.2

b:调节因子，调整字段长度对的得分的影响，默认0.75

具体的公式同样不展示了，自行查询吧。

ElasticSearch

一. 核心概念

  • Cluster集群：一个集群有多个节点
  • Node节点，服务单元
  • Shard分片，对索引做水平拆分，分片在索引创建的时候就要确定，一旦确定就不能更改。
  • Replicas副本，每个分片都有主分片以及可能的副本分片
  • Index索引：索引是集群内的唯一标识，类似于数据库
  • Tpye类别，索引内的逻辑分区，唯一标志。在es7.x后被移除。
  • Document文档，一条正向数据
  • Setting：对集群中索引的定义config
  • Mapping：类似数据库中表结构的信息，用于定义索引中Field字段的存储类型，分词方式等。es是能自动根据数据格式进行识别，但是也可以进行手动设置，索引的mapping一旦创捷且存了数据就不可修改了。
  • Analyzer：字段的分词定义，通常由tokenizer，几个Filter组成。默认的Analyzer则由标准的分词tokenizer和三个filter:StandardTokenFilter、LowerCaseTokenFilter、StopTokenFilter。

ES的节点分类如下：

  • 主节点masterNode:主节点负责创建、删除索引、分配分片、监控集群其他节点分配状态。配置node.master=true即可

    用户的请求可以发到任一节点，并由该节点进行分发和归并。并不需要es的主节点做转发。

  • 数据节点DataNode：负责存储数据即相关操作，如索引数据的增删改查聚合等。对性能要求较大。配置node.data=true即可
  • 客户端节点ClientNode：只负责请求分发归并，任意一个类型的节点都可以完成，单独增加是为了提高并发性。配置node.master=false;node.data=false即可
  • 部落节点TribeNode，用于协同多个集群，es7.0后移除
  • 协调节点CoordinatingNode：不是真正的节点，而是一种角色，集群中任何节点都可以当协调节点的角色，即负责处理分发和归并，无法配置。

集群状态有三种：

  • Green绿色，主分片和副本分片都可正常工作。
  • Yellow黄色，主分片可用，副本分片存在不可用状态。
  • Red红色，至少一个主分片及全部副本分片不可用，此时查询请求可以返回部分结果，涉及该分片的写入请款将报错。

二. 3C和脑裂

2.1 3C

3C,即一致性(Consistency)、并发性(Concurrency)和共识性(Consensus)

  • 一致性： es的写一致性通过控制校验多数分片是否可写，分为:one主分片可用即可、All所有分片均需要可用、Quorun大多数分片可用即可；此外es保证读写一致性的方式是可以通过设置sync来使主分片和备份分片在写入时进行强同步。
  • 并发性：es通过乐观锁来控制主分片写入后同步各个备份分片的无序性的。
  • 共识性：es通过自研的zen discovery来实现共识算法，大致是以Gossip的形式实现了单播

对于zen discovery ，有兴趣的读者可以自行查阅资料。

2.2 脑裂

脑裂现象就是，由于可能存在主节点因为网络原因短暂失去通信，而选举产生了另一个新的主节点。多个主节点的情形。

三. 事务日志

事务日志主要是用来解决es的内存与磁盘写入存在准实时不一致的问题。当数据在内存中没有持久化到磁盘时如果出现意外情况，则会出现数据丢失。因此es添加了事务日志(Translog)来记录还没有持久化到磁盘之间的数据。

es在写数据到内存的同时，会追加一份数据到事务日志当中。

如图:es写索引的过程经历几个步骤：

  1. es写入jvm内存中，同时追加数据到事务日志中。
  2. 触发refresh，将jvm内存写入文件缓冲区，生成一个新的段，并启用对该文件的读。同时清空jvm内存继续写数据（触发refresh的条件往往默认段刷新时间到阈值或者内存积压达到阈值）
  3. 触发fsync，先将jvm内存内数据写入文件缓冲区，清空内存，再将文件缓存区的段刷入硬盘。（触发刷盘操作往往是因为日志数据达到阈值或者刷盘时间达到阈值）
  4. 生成提交点。
  5. 清空事务日志。

四. 集群中的写索引流程

es的节点及分片例子如下图所示：

当es的写请求发出时，该请求会被发送到其中一个节点（协调节点），随后：

  1. 节点对写入的文档进行分片路由，确认将要写入那个分片pn中。
  2. 该节点将写入请求转发至pn主分片所在的节点中。
  3. pn主分片所在节点写入索引，如果成功，则异步/同步（视配置而定）请求其余副本分片执行写操作。
  4. 如果是同步分发副本分片，则等待副本分片均写入完成再返回协调节点写入成功的结果，协调节点返回写入请求结果。

分片不可修改的原因就在于路由。

五. 集群中的查询流程

查询部分可以说是重点部分了

查询部分可以分为根据routing id 查询和普通查询

5.1 routing id查询

routing id查询主要是根据文档的主键_id来查询（通常都是docId）

  1. 请求打到某一个节点，则该节点成为协调节点。
  2. 该协调节点通过进行分片路由，找出对应分片（主分片或副本分片，视轮询而定）所在的节点，将请求转发过去。
  3. 该节点进行查询，并返回结果给协调节点，协调节点返回请求。
5.2 普通查询
  1. 请求达到某一个节点时，则该节点称为协调节点
  2. 该协调节点创建一个from+size的优先队列，然后将请求分发至其余各个分片中。
  3. 各个分片均进行查询，得到对应的docId以及score，放入本地的from+size的优先队列中。
  4. 各个分片返回队列中的docId和score到协调节点中，由协调节点做多个结果的归并，最后放入自己的优先队列中。
  5. 协调节点首先找到需要返回的size条数据，然后根据docId通过路由分发到对应的分片中进行查询，归并查询结果并返回。

根据这种查询方式我们知道，我们应该尽量避免Deep Pagination（深翻），而应该采用scroll滚动。深翻中from数据越大导致浪费的查询资源就越多，性能下滑就越多。

scan扫描和scroll滚屏

scroll滚屏策略作为对深翻的一个补充,他的大致思路是对于分页查询的数据,一次性取回全量数据,并构建快照视图来进行查询.

scan指的是查询方式,即查询的时候不对结果进行排序和评分,可以通过search_type=scan来设置

其余tips

scroll查询

由于scroll会生成快照,因此段合并时不会立即删除小段,这意味着消耗更多的文件句柄.因此在查询过程中可以明确对哪些不用的scroll_id及时删除

查询时的路由

我们知道,查询时路由有两种,一种是查询语句中包含routing信息,可以直接从指定分片中进行查询;

还有一种则是分为Scatter分发,Gather收集两个过程,协调节点把请求分发到各个分片上执行,并收集返回结果.

对于普通语句,可以尝试将频率大的字段进行routing设计,其中routing算法需要保证均匀分片

对于普通语句的路由策略我们可以通过SearchType控制:

  • QUERY_THEN_FETCH:指的是第一次查询返回结果只返回docId和score,由协调节点排序后再依次去各个分片处获取.好处是节省带宽
  • QUERY_AND_FETCH: 指的第一次即返回docId,score和全量字段,由协调节点排序后直接返回客户端,节省io时间,但是许多无用消耗了带宽.
  • DFS_QUERY_THEN_FETCH:在QUERY_THEN_FETCH的基础上多了一个全局的词频计算,而不仅是分片的词频.打分更精确,但是性能更差一些
  • DFS_QUERY_AND_FETCH:与上类似.
  • 角色隔离:由于默认配置中master节点也干了data的活,可能有时候影响整个集群的性能.因此通过角色隔离,将master和data角色区分开.使用配置较低的虚拟机为master
  • 避免脑裂:上述所属由于网络原因可能回导致集群中出现2个以上deaster,导致数据丢失,为防止这种情况,我们需要配置参数discovery.zen.minimum_master_nodes = (master_eligible_nodes/2)+1

    discovery.zen.minimum_master_nodes表示选举主节点时参与选举的候选节点数(默认是1).
    master_eligible_nodes表示master集群的主节点数.

  • 还有一个tips: 在使用es时出于es对内存和缓存需要性能的考虑,作者是建议关掉swap虚拟内存空间的.

书中有一个很有点在这里提一下:
堆内存最好不要超过32GB(为什么),因为在目前普遍的64位操作系统中,内存寻址范围变大,从而使得内存指针本身也变大了,有更大的内存空间会浪费在指针本身上,同时指针在主存和缓存间移动时也会占更大的贷款.
Java在这里使用了内存压缩技术(Compressed Oops),他使得内存指针表示的是内存的偏移量而不是精确位置了,以每8个字节保存为一个引用,这样就可以以32位的寻址空间查询32GB的内存空间了,但是一旦对内存超出32G时,指针就会切换回大指针从而浪费内存**.

事实上 当内存达到40-50GB的时候才相当于压缩技术时的32GB
