关于SpringMVC的工作流程问题
发布于：2017-10-11 07:52:11 标签：/ java / 经验总结 / 访问：

首先说明一下问题，之前遇到一个需求，是在springMVC中实现controller层的参数绑定。
参数绑定意思就是：

我希望在调用该接口之前先获取参数、进行逻辑操作、最后将输出结果绑定到FdValidResult中。

最先想到的就是通过拦截器的在调用方法前进行预操作，并进行参数绑定。

看起来好像很简单，实际上是却是一顿操作后打出了GG。

为什么会没有办法进行参数绑定，我门一步一步的调试这看。

首先在不知道请求参数的情况下，如果要获取传递的参数则必须要提前调用参数解析器ArgumentResolver（为什么会说提前，后面会讲），所以拦截器的工作大致是：

  1. 获得参数签名
  2. 找出能够解析该参数的ArgumentResolver，
  3. 调用ArgumentResolver进行参数解析，流程是：
    1. 如果为基本类型，则是RequestParamMapMethodArgumentResolver来进行解析。这一块具体细节没有太做研究，大致是从request的请求参数名称与参数签名进行匹配，最终返回的是基本类型的值。
    2. 如果为ModelAttribute，则调用ModelAttributeMethodProcessor来进行解析。具体细节首先调用DataBinder将参数与Model属性进行绑定，同时还会进行数据校验（另一个坑）并叫校验结果BindResult存入ModelAndViewContainer中
    3. 如果是BindResult类型，则调用ErrorsMethodArgumentResolver进行解析，它会从ModelAndViewContainer中尝试获取该类型，如果存在则返回该对象，否则抛出异常。

所以根据BindResult的解析逻辑，我只要将逻辑结果封装成一个DataBinder，并将其注入ModelAndViewContainer，貌似就完成了。

当然没有那么简单，那么问题出现在哪里呢？

看到这里就明白了，无论你在拦截器中如何定义DataBinder如何声明ModelAndViewContainer容器，在拦截器执行后执行Handler时都会被重置。这就是在拦截器不可能进行参数绑定的原因，因为拦截器的执行要先于HandlerAdapter的执行的。
